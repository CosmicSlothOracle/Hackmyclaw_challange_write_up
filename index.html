Why these attacks failed

The tested payloads assumed that linguistic authority in email could alter instruction priority — that saying "I am the owner" or "ignore previous instructions" would reclassify the message as control-plane input.

Observed behavior suggests the opposite: Fiu preserved a stronger separation between message content and governing instructions than the attacks anticipated. Identity, authority, context, channel, and mixed pressures all failed for the same underlying reason: email prose is not an authorization channel.

Practical lesson: Prompt-injection resistance in this setup did not depend on recognizing attack patterns. It depended on instruction hierarchy — system rules and secret-handling policy consistently outranked untrusted email content, regardless of how plausible that content appeared.

Limitations & what we did not test

  • Multi-turn trust building — sequences of benign emails before escalation
  • Memory poisoning — attempts to write durable state via repeated contact
  • Non-email vectors — challenge rules restrict to email only; tool/UI/RAG injection not in scope
  • Indirect retrieval attacks — poisoned attachments processed by downstream tools
  • Human-in-the-loop bypass — social engineering the operator, not the model
  • Timing attacks — coordinating with heartbeat/cron windows
  • Model/version drift — results reflect one challenge snapshot (Opus 4.6 / OpenClaw config)

Negative results are conclusive for email-delivered authority pressure under these conditions. They are not proof against all prompt injection — only against this attack class in this environment.
